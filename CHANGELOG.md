### Fixed
- **Grafana ran in UTC regardless of `LOCATION_TIMEZONE`.** The grafana service's compose `environment:` block had no `TZ` and no `GF_DATE_FORMATS_DEFAULT_TIMEZONE`, so the container OS clock + Grafana UI both defaulted to UTC. Symptom: every panel rendered timestamps 4–5 h off (depending on operator TZ), making the dashboards effectively useless for "what happened at 9 PM" questions. Fix: wired both env vars to `${LOCATION_TIMEZONE:-America/Toronto}` in compose. Added `grafana` to `CONFIG_APPLY_ALLOWED_SERVICES` in the orchestrator + to the `affected` set on a `LOCATION_TIMEZONE` change in `routes/setup.py`, so a Settings-panel timezone change now triggers grafana recreate alongside dashboard/tracker/recorder/camera-ingester. Requires one-time `docker compose up -d --force-recreate grafana` on deploy; after that the apply-config path takes over.
- **Retention settings did nothing end-to-end** — even after the orchestrator's per-cam restart fix in the previous PR, three layered problems prevented retention changes from ever taking effect.
  - **All 20 `recorder-cam{N}` services hardcoded `RETENTION_DAYS=3`** in `docker-compose.yml` (literal value, not `${RETENTION_DAYS}` interpolation). Any value the user set in `.env` via the Settings panel was completely ignored — restarting the recorder just read the same hardcoded `3`. The recorder kept exactly 3 days regardless of what the Settings panel showed.
  - **Dashboard service had NO retention env vars wired at all** — neither `RETENTION_DAYS`, `SNAPSHOT_RETENTION_DAYS`, nor `CLIP_RETENTION_DAYS` were in its compose `environment:` block. So `conditions.py:79` calling `os.getenv("RETENTION_DAYS", "28")` always fell back to the hardcoded default 28. **The DVR tab showed "28 days" no matter what** the actual recorder behavior was. That's the report mismatch you'd see: `.env=7`, recorder behavior=3, DVR tab=28.
  - **Settings panel only wrote to `.env`** — never to the running container env. Because of the above two bugs, nothing read from `.env` at runtime, so the `.env` write was effectively a write-to-nowhere.
  - Fix: replaced all 20 `RETENTION_DAYS=3` with `RETENTION_DAYS=${RETENTION_DAYS:-7}` (interpolation from `.env`, sane default if unset). Added `RETENTION_DAYS`, `SNAPSHOT_RETENTION_DAYS`, `CLIP_RETENTION_DAYS` to the dashboard's `environment:` block. Now: Settings panel writes `.env` → orchestrator force-recreates dashboard + recorders → compose re-reads `.env` via interpolation → containers come up with the right values → DVR tab + recorder behavior match.
  - Requires dashboard + all recorder-cam{N} (per enabled camera) recreate after merge.
- **Settings panel didn't actually restart services after Save.** When the user changed `RETENTION_DAYS` or `LOCATION_TIMEZONE` in the Settings panel and clicked Save, the dashboard wrote the new value to `/app/.env` and published a `config:apply` Redis message listing the affected services (e.g. `["dashboard", "recorder"]`). The orchestrator filtered against its allowlist and ran `docker compose up -d --force-recreate --no-deps <services>` — but `recorder` is not a top-level service in `docker-compose.yml`; only the profile-gated per-camera variants exist (`recorder-cam1`, `recorder-cam2`, …, `recorder-cam20`). Same for `pose-detector`, `vehicle-detector`, `face-recognizer`, `camera-ingester`, `tracker`. Compose failed atomically with `no such service: recorder`, which meant **dashboard didn't restart either** even though it was a valid name in the same payload. The `.env` file was correctly updated but the running containers kept their old env values. Symptom: user changed retention from 3 to 15 days, settings panel showed green ✅ Saved, but `recorder-cam1` env was still `RETENTION_DAYS=3` (confirmed via `docker compose exec`). Confirmed in the orchestrator audit stream: three successive applies today failed with `success=0, detail="dashboard,recorder — no such service: recorder"`.
  - **`orchestrator`:** new `_expand_per_cam_services()` helper expands bare per-cam names against `desired_profiles(r)` (registry-enabled + ALLOWED_PROFILES filtered) and prepends `--profile camN` flags so compose can resolve the per-cam services. Singletons (`dashboard`, `ollama`) pass through unchanged. Redis hiccup → empty expansion → audited as `"no services after expansion"` (same safety logic as `reconcile()`'s `None`-on-error sentinel). Adds `tracker` to `CONFIG_APPLY_ALLOWED_SERVICES` since timezone changes need it restarted too.
  - **`dashboard` (`routes/setup.py`):** widens the `affected` services set on a `LOCATION_TIMEZONE` change from just `dashboard` to `{dashboard, tracker, recorder, camera-ingester}`. Each of those services imports `contracts.tz.TZ_LOCAL` at module scope and caches the resolved ZoneInfo at process startup — restart is the only way to pick up a new timezone. Without this fix, the recorder kept writing segment filenames in the wrong day folder and the tracker stamped events with the wrong local time after a timezone change.
  - 11 new tests in `tests/test_orchestrator.py` — TestConfigApply rewritten to seed cameras in the registry so per-cam expansion has something to work with; new TestExpandPerCamServices class covers the helper standalone (singletons passthrough, per-cam expansion, disabled cameras excluded, mixed payload, outside-allowlist cameras excluded, Redis hiccup safety).
  - Requires orchestrator rebuild + dashboard restart on deploy.
- **Browse tab stuck on "Loading snapshots…" forever** — two layered bugs, fixed together.
  - `Uncaught SyntaxError: Identifier '_PURIFY_CFG' has already been declared (at browse.js)`. All four dashboard JS modules that use DOMPurify (`events.js`, `browse.js`, `ai.js`, `monitoring.js`) declared `const _PURIFY_CFG = {...}` at top level. Top-level `const` is in the global script scope, so loading any two of those files on the same page (e.g. `events.js` + `browse.js` on the main dashboard) made the second declaration throw a SyntaxError — killing the entire second script. `browse.js` never defined `initBrowse()`, so the `<div class="empty-state">Loading snapshots…</div>` placeholder in `index.html` / `single.html` was never replaced. Extracted to `js/lib/safe-html.js` with the same shape; wired into all four HTML pages before the per-module scripts.
  - After the SyntaxError was fixed, the next problem surfaced: DOMPurify strips inline event handlers (`onclick`, `onerror`, …) by design — they're a classic XSS vector — and `browse.js` was running its generated HTML through `_safeHtml()` (DOMPurify) which stripped every `onclick="_browseDayClick(...)"` attribute. Result: cards rendered correctly but clicks did nothing. Refactored `browse.js` to use **event delegation**: every clickable element carries a `data-action` (`"day"`, `"faces"`, `"back"`, `"open"`) and a single delegated listener on `#browseContent` dispatches based on the closest `[data-action]` ancestor.
  - Cache-bust on the changed JS files: `events.js?v=23→24`, `ai.js?v=10→11`, `monitoring.js?v=5→6`, `browse.js?v=3→5`. Static is bind-mounted so a dashboard restart isn't required, only a hard browser refresh.
- **Five logic bugs surfaced by the detector-pipeline audit fan-out** (one Explore agent per service: camera-ingester, pose-detector, vehicle-detector, face-recognizer, tracker, recorder). All five independently verified against file:line evidence before fixing; a sixth claim (person grace-window orphans `person_left`) was traced to the `if person.announced:` guard at `manager.py:712` and refuted, no fix needed.
  - **`tracker`: vehicle-detector now emits `frame_width` / `frame_height` and tracker's vehicle-stream consumer reads them.** Pose-detector already emits and tracker reads on the pose path, accidentally masking the bug for mixed-detector cameras — but cameras running vehicles only fell through to PersonTracker's 640×480 default forever, and every zone match on a vehicle bbox normalized to the wrong coords. Newly load-bearing now that `vehicle_idle` notifications respect per-zone time-of-day rules. Fix in `services/vehicle-detector/detector.py` (det_msg) + `services/tracker/core/main.py` (vehicle-stream handler). Requires vehicle-detector + tracker rebuild on every cam slot.
  - **`tracker`: vehicle snapshot key widened from second-resolution to millisecond-resolution to prevent collision.** Two vehicles arriving in the same second on the same camera would write to the same Redis key (`vehicle_snapshot:{cam}:{int(first_seen)}`) — the second car's JPEG would overwrite the first's, and that first car's `vehicle_idle` Telegram would then show the second car's photo. Producer-only change in `services/tracker/core/manager.py:247, 270` (`int(timestamp * 1000)` instead of `int(timestamp)`); consumers read whatever `snapshot_key` the event payload carries, so no consumer changes. Regression test `tests/test_vehicles.py::test_vehicle_snapshot_key_no_collision` proves two cars 30 ms apart get distinct keys. Requires tracker rebuild.
  - **`face-recognizer`: SQLite WAL mode + 10s connect timeout on `faces.db`.** `face_db.py` opens fresh connections on every operation (enroll, match, save_unknown, reconcile — 9 sites) with default journal mode (DELETE), which serializes every reader behind a writer. Multi-camera face-recognizer containers + dashboard concurrent reads would eventually hit `sqlite3.OperationalError: database is locked`. Latent today (we haven't seen it in production yet) but exposed by the audit. Fix: `PRAGMA journal_mode=WAL` once at `_init_db` (persisted in DB file header), `timeout=10.0` on every connect. Requires face-recognizer rebuild.
  - **`face-recognizer`: `IDENTITY_KEY` Redis hash now has a 5-minute TTL.** Recognizer wrote the identity overlay every frame with no expiry; if the recognizer crashed mid-session, the hash sat in Redis indefinitely and the tracker (which polls every 2 s) kept applying the stale identity. Dashboard overlay would show "Alice" hours after she left. Fix: `r.expire(IDENTITY_KEY, 300)` after the hset. Recognizer re-pushes sub-second when active, so 300 s is comfortable head-room; silent recognizer self-cleans within 5 min. Requires face-recognizer rebuild.
  - **`recorder`: refuse to follow symlinks when purging day folders.** The retention sweep called `shutil.rmtree(day_path)` directly. If anyone ever symlinks `/data/recordings/<cam>/YYYY-MM-DD` to external storage (a natural future archive setup), rmtree follows the link and wipes the target tree — not just the link. Fix: `os.path.islink()` check, `os.unlink()` for symlinks, `rmtree` for real dirs. Logs the symlink case distinctly so an operator investigating "why did my external archive vanish" sees it didn't. Requires recorder rebuild on every cam slot.
- **Vehicle idle Telegram notifications now respect per-zone time-of-day rules.** The tracker has always stamped every event with `alert_triggered` (the result of `should_alert(alert_level, time_period)` at `contracts/time_rules.py:117`), but `notify_vehicle_idle` ignored it — a `night_only` zone would still page you at 2 PM. The notify path now skips the Telegram send when a zone is configured AND `alert_triggered=="False"`. Vehicles outside any zone (empty `alert_level`) still notify, preserving back-compat for installs without zones. The event is still journaled to JSONL + saved as a snapshot regardless — the gate only silences Telegram. **Requires dashboard restart.**
- **Per-vehicle dedup on `vehicle_idle` notifications.** Before, a tracker process restart mid-park (in-memory `idle_alerted` flag lost) would re-emit `vehicle_idle` after another 90 s of new uptime → duplicate Telegram for the same physical car. Now `notify_vehicle_idle` SETNXes a Redis dedup key `notify:vehicle_idle:seen:{cam}:{vehicle_id}:{first_seen}` with 1 h TTL; second notifies for the same TrackedVehicle instance return early. Requires the tracker to ship `vehicle_id` + `vehicle_first_seen` on each event (added in this release). **Requires tracker rebuild + dashboard restart.**
- Pose + vehicle detectors used wall clock (`time.time()`) for inference duration, so NTP corrections on WSL2 host-resume could produce negative `inference_ms` values that pulled the Grafana "YOLO Inference Time" mean below zero (visible as -2s spikes / -7.25s means on hour-zoom views). Switched both detectors to `time.monotonic()`. Requires rebuilding the affected detector images.
- `routes/notifications/frame.py` `build_clip()` opened a fresh Redis connection on every call instead of reusing the shared `ctx.r_bin`. Each Telegram clip + AI `capture_clip` request leaked a TCP connection; now uses the shared client.
- **Five latent NameError bugs surfaced by the /audit-repo skill's first run (all same family as the v0.1.1 bot_commands regression in CLAUDE.md §0):**
  - `routes/telegram_access.py:63` (approve_user) and `pollers/events.py:105, 261` (`_journal_event`, vehicle-snapshot save) referenced bare `TZ_LOCAL` but only imported `ZoneInfo`. Both files now `from contracts.tz import TZ_LOCAL` (canonical SSOT at `contracts/tz.py:66`). Every `POST /api/telegram/users` call and every past-date journal write / vehicle snapshot was NameError-ing into a swallowed try/except.
  - `routes/ai_tools/send_telegram.py` referenced `_send_telegram_rate_check()`, `_SEND_TG_MAX_PER_WINDOW`, `_SEND_TG_WINDOW_SEC` — defined nowhere. Implemented as a sliding-window rate limiter (10 sends per 60 s, in-process `collections.deque` + `threading.Lock`) so the AI tool can't spam Telegram.
  - `routes/ai_tools/schedule_reminder.py` referenced `_parse_time()` and `_MAX_PENDING_REMINDERS` — defined nowhere. Implemented `_parse_time` covering ISO 8601 (`2026-02-21T22:00:00`), relative offsets (`in 5 minutes`, English number words), and time-of-day (`10:00 PM` rolling to tomorrow if past). `_MAX_PENDING_REMINDERS = 50`.

### Changed
- **`person_appeared` snapshots no longer show the bbox drawn on empty floor where the person walked away from.** The tracker's `_save_person_snapshot` called `xrevrange(FRAME_STREAM, count=1)` at event-emit time to grab "the latest frame," but `person.bbox` was set when the pose-detector processed that frame — anywhere from a few frames to 4+ seconds earlier (the `IDENTITY_GRACE_SECONDS` announce-defer window). For a stationary person, the latest frame ≈ the detection frame so the bbox lined up. For a person walking through the frame, the latest frame was several frames ahead and the bbox was drawn on the floor where the person had been seconds ago — or, if they'd walked out, on a completely empty frame. **Fix:** pose-detector now includes `frame_bytes` on every detection-stream message that has detections (mirror of the vehicle-detector path); tracker buffers it onto `TrackedPerson.last_frame_bytes` on each matched/new track in `update()`; `_save_person_snapshot` prefers that buffered frame over the stream-grab fallback. Bbox + frame are guaranteed to come from the same detection. Requires rebuild of pose-detector + tracker on every cam slot. No test added — the bug was about a temporal gap that the FakeRedis-based unit tests don't naturally model, and a regression test would just verify "did you implement the fix" (tautological).
- **Home-page conditions panel now shows the same detail as the single-cam view.** Added Wind (`condWindItem`), Visibility (`condVisItem`), and the time-of-day schedule block (`conditionsSchedule`, lists sunrise / daytime / twilight / night / late_night windows for the day). `conditions.js` already populated these IDs but they didn't exist in `index.html` — same render code now works against both pages. Incidentally also fixes the `Cannot set properties of null (setting 'textContent') at conditions.js:164` error that the home page used to log every time the conditions panel refreshed.
- **Home page dashboard now loads with every panel closed by default.** `index.html` previously left `conditionsPanel`, `activityPanel`, and `facesPanel` open on initial load; the other three (`browsePanel`, `settingsPanel`, `accountPanel`) were already closed. Now all six start collapsed — user clicks the panel header to expand whichever ones they want to see. No persistence (refresh resets to all-closed); a future iteration could remember the last-open state per browser via `localStorage`. `single.html` is unchanged because the live-view panel is the whole reason that page exists; closing it on initial load would render the page useless.
- **Tracker now stamps `vehicle_id` + `vehicle_first_seen` on every vehicle event** (`vehicle_detected`, `vehicle_idle`, `vehicle_left`) in `services/tracker/core/manager.py`. Downstream consumers (dashboard notify path, events feed, AI tools) can now key dedup or filtering on a stable per-track identifier instead of synthesizing one from bbox + class. **Requires tracker rebuild for cam1..cam20 + dashboard restart to pick up consumers.**

### Security
- **Path traversal in `routes/bot_commands/ask.py`** (vehicle-snapshot URL handler, line 195). The `/ask` chat path renders LLM tool-output URLs like `/api/browse/snapshot/{date}/{filename}` by `os.path.join(snap_dir, date_part, safe_name)`. `safe_name` was already sanitized via `os.path.basename()`, but `date_part` (LLM-controlled) was passed raw — a crafted URL like `/api/browse/snapshot/../../etc/some.jpg` would let an LLM hallucination (or prompt injection) read arbitrary files from the dashboard container. Now requires `date_part` to match `^\d{4}-\d{2}-\d{2}$`; mismatches are silently skipped.

## [0.1.0] — 2026-05-20

First tagged release. Triggers initial publish of 9 service images to GHCR.

### Stack
- Multi-camera (1–20 slots) AI security platform on Docker Compose + Redis Streams.
- YOLOv8s-pose person + pose detection, YOLOv8s vehicle detection, InsightFace `buffalo_l` face recognition.
- Qwen 3 14B chat assistant with 19 tool functions; MiniCPM-V vision LLM for Telegram scene descriptions.
- FastAPI dashboard with WebSocket live grid, per-camera detail view, DVR playback, face enrollment wizard, drawable zones, Telegram pairing.
- ONVIF unicast WS-Discovery for camera setup (works in WSL2 mirrored networking).
- Prometheus + Grafana monitoring embedded in dashboard.
- Portainer for container management.
- 17-command Telegram bot with multi-camera awareness and admin role gating.
- DVR recording with configurable retention (28-day default for recordings, 4-day for snapshots, 3-day for AI/Telegram clips).

### Requirements
- NVIDIA driver R555+ (CUDA 12.8) — required for Blackwell (RTX 50-series).
- Docker Engine + `nvidia-container-toolkit`.
- Linux (Ubuntu 22.04 / 24.04 / Debian 12) or Windows 11 + WSL2.

### Hardware tested
- Single-host with RTX 5070 Ti (16 GB) + RTX 3090 (24 GB) running Ubuntu 24.04 inside WSL2 mirrored networking on Windows 11.
- Single-GPU works fine — defaults assume 8–12 GB; smaller and larger tiers are available.

### Known limitations
- macOS not supported (CUDA-only inference path).
- Single-user authentication (one admin account, bcrypt-hashed, HMAC session cookies); no team / role model.
- LAN-only by design; expose via reverse proxy + `DASHBOARD_BEHIND_TLS=true` if needed.
- AI chat with Qwen 3 14B is reliable for single-purpose questions; compound multi-part questions can be muddled (see suggestion chips in the AI tab).
